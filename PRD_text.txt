1. Executive summary
WP Admin Nav Designer is a WordPress plugin that replaces the default WP admin sidebar and top bar with modern, SaaS-inspired navigation templates. Users pick from 10 pre-built templates, set a brand color, and the entire admin nav updates instantly — no code required.
The plugin is deliberately lightweight. It stores all configuration in the WordPress options table (no custom DB tables), exposes a standard WP REST API, and uses React with local useState (no Redux). This keeps the codebase lean, the install footprint small, and compatibility with managed hosting maximized.

Existing plugins solve structural problems (hiding and reordering menu items) but none deliver a modern, cohesive visual redesign of both the sidebar and top bar together.

3. Architecture overview
The plugin is intentionally simple. The architectural decisions below were made explicitly — they are not defaults to revisit without a compelling reason.

3.1 Architecture decision record
Decision
Choice
Why
Trade-off
Storage
WordPress options (JSON)
No custom table migration, no schema version management, no Eloquent ORM. Works on all hosts instantly.
Not suited for large relational datasets — acceptable since template config is small.
REST API
Standard WP REST API
register_rest_route() is universally understood, auto-documented, and tested. No custom router dependency.
Less flexible than a custom router for complex param binding — acceptable for this feature scope.
Frontend state
React useState
No Redux overhead. Settings page is a single admin screen with simple form state — local state is sufficient.
Pro extension will need a lightweight bridge if it needs to share state — use window.SaasMenu instead.
Auth
current_user_can('manage_options')
All plugin configuration is admin-only. A single capability check is correct and simple.
Coarser than per-feature permissions — acceptable since all features are admin-only.
DB tables
None — wp_options only
Avoids activation/migration complexity. Single option key saasmenu_settings stores the full config JSON.
Cannot query config relationally — acceptable since config is always loaded as a whole.
CSS prefix
saasmenu-
Avoids collision with WP core, other plugins, and the banned class list.
Slightly more verbose than pm- but prevents any naming conflicts.

3.3 Free / Pro extension bridge
The plugin exposes a window.SaasMenu object from index.jsx. Pro attaches at runtime. This is an optional extension surface — the free plugin works fully without Pro installed.

// views/assets/src/index.jsx
window.SaasMenu = {
  registerSlot,    // Pro fills named slots
  registerFilter,  // Pro replaces filtered sections
  Slot,            // Free declares <Slot name='...' />
  useFilter,       // Free declares useFilter('...', fallback)
  components: { TemplateSelector, ColorCustomizer, MenuPreview },
  utils:      { sanitize },
}

Bridge rules
Adding a new key to window.SaasMenu is additive and safe. Removing or renaming any key is a BREAKING change that requires coordinating with the Pro repo. Free files must never import from ../pm-pro/. Pro must never import from wp-admin-nav-designer/views/assets/src/ directly — all access goes through window.SaasMenu.

4.2 REST API
4.2.1 Base URL and auth
All endpoints use the standard WordPress REST API — register_rest_route() only. No custom router.

Base URL: /wp-json/saasmenu/v1/

Auth on every request:
  Header:  X-WP-Nonce: SaasMenu_Vars.permission
  Param:   is_admin: SaasMenu_Vars.is_admin   (hint, not a trust boundary)

Permission callback on all mutation endpoints:
  'permission_callback' => function() {
      return current_user_can('manage_options');
  }

4.2.2 Endpoints
Method + path

Description

4.2.4 Endpoint registration pattern
All routes registered in routes/templates.php, auto-loaded by bootstrap/loaders.php.

<?php
// routes/templates.php
add_action( 'rest_api_init', function() {

    register_rest_route( 'saasmenu/v1', '/templates', [
        'methods'             => 'GET',
        'callback'            => [ new Template_Controller, 'get_templates' ],
        'permission_callback' => '__return_true',
    ] );

    register_rest_route( 'saasmenu/v1', '/settings', [
        'methods'             => 'POST',
        'callback'            => [ new Template_Controller, 'save_settings' ],
        'permission_callback' => function() {
            return current_user_can( 'manage_options' );
        },
    ] );

} );

4.3 Controller pattern
<?php
namespace SaasMenu\Controllers;

class Template_Controller {

    public function save_settings( \WP_REST_Request $request ) {
        // 1. Sanitize every param at entry — never trust $request->get_params() raw
        $data = [
            'templateId'  => sanitize_text_field( $request->get_param( 'templateId' ) ),
            'brandColor'  => sanitize_hex_color( $request->get_param( 'brandColor' ) ),
            'darkMode'    => (bool) $request->get_param( 'darkMode' ),
            'density'     => sanitize_text_field( $request->get_param( 'density' ) ),
            'customLogo'  => esc_url_raw( $request->get_param( 'customLogo' ) ),
        ];

        // 2. Write to options
        update_option( 'saasmenu_settings', $data );

        // 3. Fire action after mutation
        do_action( 'saasmenu_after_save_settings', $data );

        // 4. Return JSON (wp_send_json acceptable for simple responses)
        return rest_ensure_response( [ 'success' => true, 'data' => $data ] );
    }
}

4.6 Security requirements
All mutation REST endpoints must have a permission_callback that verifies current_user_can('manage_options').
All $request->get_param() results must be sanitized before writing to the database or returning in responses.
No raw $wpdb concatenation — use update_option() / get_option() for settings; if direct SQL is ever needed, use $wpdb->prepare().
No eval(), no base64-encoded PHP, no obfuscated code — plugin must pass the WP Plugin Review Team automated scan.
License keys must never be stored in plain text — encrypt before writing to wp_options.
No API keys or tokens may be injected into SaasMenu_Vars (the JS bundle is public). Tokens stay server-side.
SVG uploads (custom logo): must run through WordPress wp_check_filetype_and_ext filter. Use enshrined/svg-sanitize if SVG support is added.
wp_unslash() must be called before any sanitize_*() on $_POST / $_REQUEST values.

5.2 JS variables injected by PHP
PHP injects a SaasMenu_Vars object via wp_localize_script() before the bundle loads. These are the only values the frontend may read from PHP — no other globals.

window.SaasMenu_Vars = {
  rest_url:    'https://site.tld/wp-json/saasmenu/v1/',
  permission:  '...',        // WP nonce — used as X-WP-Nonce header
  is_admin:    true,         // hint only — not a trust boundary
  is_pro:      false,        // controls Pro feature visibility in UI
  settings:    { ... },      // current saasmenu_settings value
  templates:   [ ... ],      // built-in template definitions array
  language: {
    locale_data: { 'saasmenu': { ... } }  // JED format for @wordpress/i18n
  }
}

5.3 API access — non-negotiable rules
All HTTP requests must go through the useApi hook. Raw fetch() calls are not permitted anywhere in the frontend codebase.

// views/assets/src/hooks/useApi.js
const useApi = () => {
  const get  = (path, params = {}) => fetch(`${SaasMenu_Vars.rest_url}${path}?${buildQueryString({ ...params, is_admin: SaasMenu_Vars.is_admin })}`, {
    headers: { 'X-WP-Nonce': SaasMenu_Vars.permission }
  }).then(r => r.json())

  const post = (path, body = {}) => fetch(`${SaasMenu_Vars.rest_url}${path}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'X-WP-Nonce': SaasMenu_Vars.permission },
    body: JSON.stringify({ ...body, is_admin: SaasMenu_Vars.is_admin })
  }).then(r => r.json())

  return { get, post }
}

7. Functional requirements
7.1 Template engine
Each template is a self-contained CSS file loaded via admin_enqueue_styles when that template is active. No PHP template files.
Template CSS must be scoped exclusively to #wpadminbar and #adminmenu selectors to prevent bleed into page content.
Template switching applies on next page load (no full live CSS injection required for v1.0 — live preview uses the /preview/css endpoint).
Deactivating the plugin must fully restore the default WP admin appearance — all template CSS is removed, no residual styles.
Templates must not override WP core JS or PHP hooks that affect functionality — visual layer only.

7.4 Brand color system
Single hex color input in the settings panel.
On input, derive: active state color, hover state (10% opacity tint), text-on-color contrast (WCAG auto-select black or white at ≥4.5:1).
Color propagates to: sidebar active highlight, top bar accent, button hovers, submenu active states, link colors within the nav.
Live preview via POST /preview/css — returns generated CSS string rendered in a preview iframe.
Color persists between template switches (brand color overrides template default).
'Reset to template default' option available in settings.
Warn user when selected color fails WCAG 2.1 AA contrast against the template's default text color.

7.5 Role-based visibility (v1.0)
Admin can show/hide any menu item per WordPress user role.
Settings panel lists all registered WP roles and all current menu items in a matrix.
Changes apply on next page load for the affected role (no cache invalidation required).
Must not conflict with other role-management plugins — visibility rules are additive CSS-only, not capability removal.
Implementation: output a scoped CSS rule that hides the menu item by its WP-generated ID for the user's role class on body.

7.6 Onboarding wizard
Step 1 — Choose a template: Grid of all templates (free tier shows Pro ones as locked). Tap to select.
Step 2 — Set brand color: Pre-populated with the active theme's primary color if detectable. Skip option available.
Step 3 — Done: Summary of applied changes + 'View your new admin' CTA.
Wizard appears once on activation. Accessible again from Settings > Nav Designer > Setup Wizard.
Skipping any step applies defaults (template: linear-dark; color: template default).
Target completion time: under 60 seconds.

7.7 Settings panel
Location: Settings > Nav Designer (wp-admin settings page).
Tabs: Templates | Sidebar | Top Bar | Colors | Roles | Advanced | License.
All settings saved via REST POST /settings with AJAX (no full page reload).
Settings stored in wp_options as saasmenu_settings (single serialized JSON array).
Export settings button (Agency plan): downloads saasmenu_settings as a JSON file.
Import settings button (Agency plan): accepts JSON file and writes to saasmenu_settings.

11. Risks and mitigations
Risk
Mitigation
WP core CSS update changes admin selectors
Scope all CSS to known stable selectors (#wpadminbar, #adminmenu, #adminmenuback). Monitor WP core Trac. Add a compatibility test that screenshots the admin and diffs vs baseline on every WP beta.
Third-party plugin conflict (white screen of death)
All CSS scoped to WP admin selectors only. Provide ?saasmenu_safe_mode=1 query param that disables template CSS for one session without requiring WP admin access.
WordPress.org review rejection
Submit per official guidelines. No encrypted code, no external API calls beyond the license server, no obfuscation. Tested with Plugin Check tool before submission.
Low free-to-Pro conversion (<5%)
A/B test upgrade prompts at month 2. Interview churned free users. Fallback: add a more compelling free-tier limitation (e.g. 1 template on free instead of 3).
License API downtime blocks Pro activations
7-day grace period on license validation failure. Plugin degrades to free-tier behavior — does not deactivate or delete settings.
window.SaasMenu contract broken after a refactor
Run /free-pro-contract-check (see SKILL.md) before merging any PR that touches index.jsx. Add a CI step that diffs window.SaasMenu shape against the last tag.
